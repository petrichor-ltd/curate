- 主題：當工具層被平台商品化，企業真正該保留什麼
- 標籤：Agent Skills、Ontology、Palantir、企業本體、資料主權、模型主權、技術壁壘、FDE、token 定價、服務價值

## 1. 原始文章

這是 7 月 17 號的一些思考與沉澱。

今天看到一篇文章，探討在 AI 大語言模型進展下（特別是針對最近的 Codex、Anthropic，甚至 Gemini 等大模型）的產業變局。在這些大模型的清洗與侵襲之下，現在甚至有一種說法：Skills 已經越來越式微，未來可能甚至不需要 Skills 了。

因為我們現在所做的這些 Skills，未來有很大的機率會被大模型公司一個反手，直接打破所有的技術壁壘。大模型公司現在完全可以刻意讓廠商去開發自己的 Skills，但充其量這些 Skills 終究只是一些點 MD 檔或資料的堆疊，相對是沒有技術壁壘的。它們很容易被衝破，大型公司只要稍微調整一下，就可以直接把這些成果拉過去。

現在反而出現一個更新的理論，叫做「本體論」（Ontology）：我們該如何 on top of 這些技術，然後直接回歸到企業的「本體」？

實際上，企業應該去針對實際業務更改核心演算法或核心大語言模型，形成一個專屬的「大語言模型離線版」。這樣做的好處，首先是資料的保密性與安全性，因為一切終究回歸到公司的本體。

在美國，Palantir 這間公司就已經完全跑通了這個商業模式，這其實就是未來的縮影與可預見的情況。

這又回到另外一個點。當時 Palantir 的 CEO 也跳出來質疑：為什麼目前主流的大語言模型公司都要以 token 來計價？他道出了一個非常直接的痛點：「你用 token 計價，是不是代表你對自己的大語言模型沒有信心？為什麼不是以服務來定價？」

比方說像 Palantir，他們就是直接提供一個服務，可以直接部署到客戶的公司。這也是 FDE（Forward Deployed Engineer）這個工程師職位鼻祖的工作模式。

這講到一個很有趣的點：為什麼要用 token 計價？

如果我們現在持續在使用 Skills，其實有點像是在落入大語言模型公司的「陽謀」：你名義上在開發自己的應用，實際上卻是在幫他們訓練模型、訓練整套系統。這終究不是終局之戰。

終局之戰終究還是要像 Palantir 這樣的公司，能夠提供真正幫助公司拓展業務的服務，而不是用陽謀的形式，讓大模型公司去竊取其他企業的技術與資料。

## 2. AI 潤飾版（200-300 字）

今天讀到 Skills 可能被更強大的模型迅速吸收時，我真正不安的，並非技術過時本身，而是辛苦累積的能力，是否一直寄放在別人的平台上。Skill 的格式或許容易複製，但企業對自身業務的理解、資料之間的關係、做決定的理由，以及最後如何把結果帶回現場，並不會因模型升級就自然長出來。Ontology 讓我重新思考：真正該守住的，也許不是一顆永遠不變的模型，而是公司理解自己、修正自己與採取行動的能力。至於 token 或服務定價，背後追問的也是同一件事：我們付費買到的是更多運算，還是真正解決了問題？我仍擔心平台依賴與資料風險，但比起把它們寫成必然的掠奪，我更想先弄清楚，哪些核心判斷不能外包。

## 3. 影片主旨稿（5-8 分鐘）

今天我看到一篇文章，裡面提出一個很直接的判斷：Skills 可能很快就不值錢了。

我讀到這裡時，第一個感覺不是反對，而是不安。因為我們現在花時間整理的流程、規則和工具，會不會在下一次模型更新後，突然變成平台原生能力？如果答案是會，那我們到底在累積什麼？

這個問題不能只用「Skills 會不會消失」來回答。Anthropic 對 Agent Skills 的官方定義，是由指令、腳本與資源組成的目錄，模型會依任務需要載入內容，也能執行其中的程式。所以，最簡單的 Skill 確實可能只是一份容易複製的說明，但比較完整的 Skill 還包含可重複執行的程式、工作流程與組織經驗。

模型愈強，有些指令會變得多餘。過去需要寫十頁規則的工作，未來也許一句話就能完成。這一層的技術壁壘很薄，我同意。

可是，格式容易被複製，不代表能力本身已經被複製。

**真正容易被拿走的，不是一份檔案，而是我們誤把檔案當成了能力本身。**

一間公司真正難以移植的東西，往往藏在日常裡：客戶為什麼在某個時刻拒絕？什麼資料可以被誰看見？一個異常發生時，誰有權停下流程？哪個指標看起來漂亮，卻會傷害長期信任？

這些內容不只是知識，也包含判斷、權限、責任和後果。把它們寫成一份 Markdown，並不代表企業已經理解自己；但完全不整理，它們又只會散落在少數人的腦中。

這也是我開始注意 Ontology 的原因。

Palantir 對 Ontology 的描述，並不是重新訓練一顆公司的專屬大模型。它更像是把企業的資料、邏輯、行動與安全規則，整理成一個人與 AI 都能理解、也能實際操作的共同世界。Palantir 的 AIP 仍可以連接商業模型、開源模型或企業既有模型。

所以，重點不是一定要擁有一顆離線大模型。更重要的是，即使底層模型換了，公司對訂單、客戶、設備、風險與決策的理解，仍然留在自己手上。

**工具會被平台收進去，但企業如何理解自己，不該跟著被收走。**

這也讓我重新看待 FDE。工程師不只坐在遠處接需求，而是進入客戶現場，理解真正卡住的問題，再把資料、軟體與工作流程接起來。它賣的不是一串 token，而是從模糊問題走到可運作結果的能力。

可是，這不代表 token 計價一定錯，也不能因為模型公司按 token 收費，就直接判定它們對產品沒有信心。

token 計價是在賣可以量測的運算投入；服務或成果定價則把價格綁在結果上。後者聽起來更負責，但結果究竟由模型、資料、流程、員工採用，還是市場環境造成，常常很難切清楚。當成果無法歸因，服務商也可能把風險溢價放進價格裡。

真正值得保留的問題是：企業花了這些錢之後，有沒有得到可驗證的改變？如果只有 token 持續增加，流程卻沒有更好，那再強的模型也只是昂貴的活動量。

資料風險也需要同樣冷靜。使用 Skill 並不自動等於替模型公司訓練。至少 Anthropic 目前對商業產品的說明，是預設不使用客戶的對話與 coding sessions 訓練模型，除非客戶主動加入相關計畫。消費者方案、商業契約、資料保存與回饋設定仍有不同，企業必須逐項確認。

我仍然認同資料主權很重要。只是如果把尚未查清楚的風險直接寫成「竊取」，警惕反而會失去準確性。我們需要的不是先相信平台，也不是先把平台定罪，而是知道資料去了哪裡、保留多久、誰能使用，以及換掉供應商時能不能帶走自己的能力。

**真正的主權，不是拒絕所有外部模型，而是換了模型之後，企業仍知道自己在做什麼。**

因此，Skills 或許不是終局，但 Ontology 也不會是一個新的萬能答案。如果企業沒有真的理解業務，只是把更多資料堆進另一套系統，那只是從 Markdown 的堆疊，換成更昂貴的名詞堆疊。

真正的本體，應該包含公司如何命名現實、如何做判斷、如何採取行動，也包含出錯時由誰承擔。這些東西需要工程師、業務、現場人員與管理者一起整理，不會由任何模型一鍵生成。

**企業的護城河，不是模型知道多少，而是企業能不能把自己的經驗變成可驗證、可修正、也可負責的行動。**

所以，今天這篇文章帶給我的，不是「從此不要再做 Skills」。它更像是一個提醒：不要只累積平台下一次更新就能收走的表面能力。

我還是會做 Skill，但會更常問：這份 Skill 背後留下了什麼？是一段容易取代的提示詞，還是一套屬於我們的判斷、資料關係與回饋循環？當模型變得更強，我真正想保留的，也許就是那個仍能回答「我們為什麼這樣做」的人與組織。

## 4. 文章版（約 1200 字）

### 我還是會做 Skills，但要留下能帶走的東西

今天讀到一個判斷：隨著 Codex、Claude、Gemini 這類模型進步，Skills 可能逐漸失去價值。模型一更新，外部整理的能力就可能被收進原生產品。我擔心的不只是單一技術過時。更深一層，是辛苦累積的東西，是否一直寄放在別人的平台上。

這個擔心有道理，卻要分清 Skill 的哪一層會被取代。Anthropic 將它定義為包含指令、腳本與資源的目錄。簡單提示詞的壁壘確實很薄，經過驗證的程式、例外與權限卻不會只因格式公開就失去價值。這讓我放下「格式被吸收，能力就全被拿走」的恐懼，卻留下更難的問題：這些判斷究竟留在誰手上？

真正脆弱的，也許不是 Skill，而是把檔案誤認成能力。一間公司的核心不只藏在文件裡。客戶為什麼拒絕、什麼資料能被誰看見、哪個例外必須由人接手、錯誤最後由誰負責，這些才是經驗形成的判斷。文件能承載它們，卻不能讓它們自動成立。

因此，Ontology 吸引我，但也要說準。Palantir 的 AIP 文件將它描述為企業資料、邏輯、行動與安全的共同表示，仍可連接不同模型。它不等於重訓一顆離線大模型，而是讓模型可以替換，企業對現實的命名、決策與權限仍能累積。

可是，換得了模型，不等於換得了服務商。若 Ontology、現場整合與成果交付都由另一個大型平台掌握，企業可能只是把能力搬進更難拆開的服務關係。若資料留在本機，卻無人能解釋它如何進入決策，主權也只停在儲存位置。真正的企業本體，至少要讓資料、流程、判斷與責任彼此連得起來，也能被驗證與修正。

token 定價也回到同一個問題。它對運算收費；服務定價則綁在結果。後者更接近客戶價值，卻不必然更誠實，因為結果也受資料、流程、員工與市場影響。服務商若包下結果，企業還可能把問題定義、回饋與變更能力一起外包。token 價格不能證明廠商沒有信心，但只看使用量，也容易把活動誤認成成果。

我最初想用「陽謀」甚至「竊取」替這份不安命名，但那把幾件事混在了一起。資料是否進入訓練，取決於產品、契約與設定。Anthropic 目前說明，其商業產品預設不使用客戶對話與 coding sessions 訓練模型，除非客戶主動參與相關計畫。這不表示風險不存在，而是外洩、留存、訓練與依賴必須分開查證。收回過度的指控，不等於警戒已經消失。

可是，今日的政策不代表企業可以放棄警覺。資料去了哪裡、保留多久？模型或服務商改變做法甚至停止服務時，能否帶走流程、評估與決策脈絡？主權不是把一切關在內網，而是依賴外部能力時，仍知道哪些判斷、資料關係與責任屬於自己。

所以，我不會因為 Skills 可能被商品化，就停止整理。若留下的只是提示詞，它本來就不該被當成壁壘；若它承載了反覆驗證的流程、例外與責任，它才是通往本體的材料。

模型會變強，工具會被吸收，新名詞也會出現。我還沒有把握 Ontology 是否就是終局，也不知道它會不會只是更昂貴的平台名稱。下一次再做一份 Skill，我想追問：如果明天格式被原生能力取代，留下的流程、例外與回饋能不能被驗證、被帶走？換掉模型與服務商後，我們還能不能說清楚自己為何這樣做？也許只有走進實作，我才知道累積的是自己的理解，還是另一份寄放在平台裡的表面能力。
